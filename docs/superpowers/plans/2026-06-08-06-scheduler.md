# Plan 6 — Scheduler: Reminders & Pending-Booking Auto-Expiry Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add the two pieces of stateless background work the overview defers to this plan: **feature 15** (reminder emails at a configurable lead time before a confirmed meeting) and **feature 14** (auto-expiry of approval-mode bookings whose hold window has elapsed). Both run on a `quarkus-scheduler` tick on *every* replica and are made multi-node-safe **without a leader, without a clustered lock, and without a Quartz JobStore** — each tick claims due rows with Postgres `SELECT … FOR UPDATE SKIP LOCKED` and marks them processed in the same transaction, so exactly one replica handles each item. This directly satisfies the overview NFR bullet *"Stateless background work"*: "each tick claims due rows via Postgres `SELECT … FOR UPDATE SKIP LOCKED` and marks them processed, so exactly one replica handles each item. No clustered lock / quartz JobStore needed — the DB row-claim is the arbiter."

**Architecture:** A new `reminder` table (Flyway **V5**) holds one unsent row per confirmed booking, due at `startUtc − leadTime`. A `@ApplicationScoped` `ReminderScheduler` observes the booking lifecycle events (Plan 3) to **create**, **recompute**, or **delete** reminder rows, and exposes one `@Scheduled(every = "60s")` tick method that:

1. **Reminder dispatch:** claims due unsent `reminder` rows via a native `SELECT … FOR UPDATE SKIP LOCKED LIMIT 50`, sets `sent_at = now()` on the claimed rows in the same transaction, and fires a CDI `ReminderDue(bookingId)` event per claimed row. Plan 4's email layer observes `ReminderDue` and applies the same invitee-fallback / owner-opt-out rules it already uses for the other booking emails.
2. **Pending auto-expiry (feature 14):** claims `PENDING` bookings whose hold has expired (`expiry = min(createdAt + holdWindow, startUtc)`) via a second `SELECT … FOR UPDATE SKIP LOCKED LIMIT 50`, flips them to `DECLINED`, and fires the existing `BookingDeclined(bookingId)` event (Plan 3). Because the overlap-exclusion guard only covers `status IN ('PENDING','CONFIRMED')`, flipping to `DECLINED` removes the row from the guard and **frees the held slot** for a new booking — at both the app level and the DB level — with no extra bookkeeping.

`SELECT … FOR UPDATE SKIP LOCKED` is the entire concurrency story: when replica A holds a row lock inside its tick transaction, replica B's identical query simply *skips* that locked row and claims the next available one. Two replicas therefore never claim the same row, so each reminder is sent once and each pending booking is declined once. We mark `sent_at` / flip `status` inside the same transaction that holds the lock, so the claim and the state change commit atomically; a crash before commit rolls back the claim and the row is retried on the next tick (at-least-once becomes effectively exactly-once because the marked row is excluded from future claims).

**Tech Stack:** Java 25, Quarkus 3.35.3, `quarkus-scheduler` (added in Task 1), Hibernate ORM Panache, Postgres, Flyway, JUnit 5. Tests run against Quarkus **Dev Services** (Testcontainers Postgres) — **Docker must be running**. Tests call the tick methods **directly** (the scheduler interval is irrelevant to correctness) and assert claim/mark/flip behavior; the cross-replica `SKIP LOCKED` guarantee is **documented, not faked** (a single-threaded test cannot reproduce two racing replicas — see Self-Review).

```sql
-- Feature 15: one pending reminder row per CONFIRMED booking, due at
-- (booking.start_utc - lead time). The scheduler tick (Plan 6) claims due
-- unsent rows with SELECT ... FOR UPDATE SKIP LOCKED and marks them sent.
-- ON DELETE CASCADE: if a booking row is ever hard-deleted, its reminders go
-- with it (cancel/decline are soft status flips, handled in the app by deleting
-- the unsent reminder explicitly — see ReminderScheduler).
CREATE TABLE reminder (
    id          BIGSERIAL    PRIMARY KEY,
    booking_id  BIGINT       NOT NULL REFERENCES booking(id) ON DELETE CASCADE,
    send_at     TIMESTAMPTZ  NOT NULL,
    kind        VARCHAR(24)  NOT NULL,          -- e.g. 'REMINDER'
    sent_at     TIMESTAMPTZ                     -- NULL = unsent
);

-- The tick scans for due, unsent reminders: WHERE sent_at IS NULL AND send_at <= now().
-- A partial index on (send_at) WHERE sent_at IS NULL keeps that claim query cheap as
-- the table accumulates already-sent rows.
CREATE INDEX idx_reminder_due ON reminder (send_at) WHERE sent_at IS NULL;
```

### Task 5: `ReminderScheduler` — create / recompute / delete reminder rows on booking events

The scheduler service that owns reminder lifecycle. This task wires the **creation** side (observe lifecycle events, insert/delete `reminder` rows). The tick methods are added in Task 6.

**Files:**
- Create: `src/main/java/com/calit/scheduler/ReminderScheduler.java`
- Modify: `src/main/resources/application.properties`
- Test: `src/test/java/com/calit/scheduler/ReminderCreationTest.java`

**Configuration:**
- `calit.reminders.lead-minutes` (int, default `1440` = 24h) — how far before `startUtc` the reminder is sent.

**Behavior contract (creation side):**
- A reminder is created when a booking becomes **CONFIRMED** — either auto-confirmed at book time (`BookingConfirmed`) or approved from PENDING (`BookingApproved`). Observe **both** events with one helper `scheduleReminder(Long bookingId)`.
- `scheduleReminder`: load the booking; compute `sendAt = booking.startUtc − leadMinutes`. **If `sendAt` is already in the past (`<= now()`), skip** (no row inserted — a booking made inside the lead window simply gets no reminder). Otherwise first `Reminder.deleteUnsentFor(bookingId)` (so we never double-schedule on re-confirm/reschedule-into-confirm) then insert one unsent `Reminder` (`kind = REMINDER`, `sentAt = null`).
- On **`BookingCancelled`** and **`BookingDeclined`**: `Reminder.deleteUnsentFor(bookingId)` — drop the future unsent reminder (already-sent rows are untouched).
- On **`BookingRescheduled`**: **recompute** — `Reminder.deleteUnsentFor(bookingId)` then `scheduleReminder(bookingId)` against the new `startUtc` (auto-type reschedules stay CONFIRMED, so the moved booking still warrants a reminder at the new time). For approval-type reschedules the booking returns to PENDING and `BookingRescheduled` is not fired (Plan 3 re-fires `BookingRequested`), so no stale reminder survives; the eventual `BookingApproved` reschedules it.
- **Transaction:** observers run synchronously in the booking's own transaction by default. To stay consistent with Plan 4's pattern (act only after the booking row is durable) these observers use `@Observes(during = TransactionPhase.AFTER_SUCCESS)` and open a fresh transaction with `QuarkusTransaction.requiringNew()` to load the booking and write the reminder row — so a reminder is scheduled only after the booking commit succeeds, and never on a transaction that later rolls back.

`src/main/java/com/calit/scheduler/ReminderScheduler.java`:

```java
package com.calit.scheduler;

import com.calit.booking.Booking;
import com.calit.booking.events.BookingApproved;
import com.calit.booking.events.BookingCancelled;
import com.calit.booking.events.BookingConfirmed;
import com.calit.booking.events.BookingDeclined;
import com.calit.booking.events.BookingRescheduled;
import io.quarkus.narayana.jta.QuarkusTransaction;
import jakarta.enterprise.context.ApplicationScoped;
import jakarta.enterprise.event.Observes;
import jakarta.enterprise.event.TransactionPhase;
import org.eclipse.microprofile.config.inject.ConfigProperty;

import java.time.Instant;
import java.time.temporal.ChronoUnit;

@ApplicationScoped
public class ReminderScheduler {

    @ConfigProperty(name = "calit.reminders.lead-minutes", defaultValue = "1440")
    int leadMinutes;

    // --- lifecycle observers (creation/recompute/delete side) ---

    /** Auto-confirmed at book time. */
    void onConfirmed(@Observes(during = TransactionPhase.AFTER_SUCCESS) BookingConfirmed e) {
        scheduleReminder(e.bookingId());
    }

    /** PENDING -> CONFIRMED via owner approval. */
    void onApproved(@Observes(during = TransactionPhase.AFTER_SUCCESS) BookingApproved e) {
        scheduleReminder(e.bookingId());
    }

    /** Cancelled by invitee. */
    void onCancelled(@Observes(during = TransactionPhase.AFTER_SUCCESS) BookingCancelled e) {
        onCancelledOrDeclined(e.bookingId());
    }

    /** Declined by owner OR auto-expired (Plan 6 expiry tick fires BookingDeclined). */
    void onDeclined(@Observes(during = TransactionPhase.AFTER_SUCCESS) BookingDeclined e) {
        onCancelledOrDeclined(e.bookingId());
    }

    /** Auto-type reschedule stays CONFIRMED at a new time: recompute the reminder. */
    void onRescheduled(@Observes(during = TransactionPhase.AFTER_SUCCESS) BookingRescheduled e) {
        scheduleReminder(e.bookingId());
    }

    /**
     * Create (or replace) the unsent reminder for a now-CONFIRMED booking at
     * (startUtc - leadMinutes). Skips if that instant is already in the past.
     * Opens its own transaction (AFTER_SUCCESS observers have no active one).
     */
    public void scheduleReminder(Long bookingId) {
        QuarkusTransaction.requiringNew().run(() -> {
            Booking booking = Booking.findById(bookingId);
            if (booking == null) {
                return;
            }
            Instant sendAt = booking.startUtc.minus(leadMinutes, ChronoUnit.MINUTES);
            // Booking made inside the lead window: nothing to remind about ahead of time.
            if (!sendAt.isAfter(Instant.now())) {
                return;
            }
            // Never double-schedule (re-confirm / reschedule recompute).
            Reminder.deleteUnsentFor(bookingId);
            Reminder r = new Reminder();
            r.bookingId = bookingId;
            r.sendAt = sendAt;
            r.kind = Reminder.KIND_REMINDER;
            r.sentAt = null;
            r.persist();
        });
    }

    /** Drop the future unsent reminder for a cancelled/declined/expired booking. */
    public void onCancelledOrDeclined(Long bookingId) {
        QuarkusTransaction.requiringNew().run(() -> Reminder.deleteUnsentFor(bookingId));
    }
}
```

### Task 6: The tick — claim due reminders with `SELECT … FOR UPDATE SKIP LOCKED`, mark sent, fire `ReminderDue`

The core multi-node-safe dispatch. Adds the `@Scheduled(every = "60s")` reminder tick plus its claim/mark logic to `ReminderScheduler`.

**Files:**
- Modify: `src/main/java/com/calit/scheduler/ReminderScheduler.java`
- Test: `src/test/java/com/calit/scheduler/ReminderTickTest.java`

**Behavior contract for `dispatchDueReminders()`:**
1. In a single transaction, run a **native** claim query:
   ```sql
   SELECT id FROM reminder
   WHERE sent_at IS NULL AND send_at <= now()
   ORDER BY send_at
   FOR UPDATE SKIP LOCKED
   LIMIT 50
   ```
   `FOR UPDATE` locks each returned row for the lifetime of this transaction; `SKIP LOCKED` means a *second* replica running the identical query at the same moment **skips** any row this replica has already locked and claims different rows. **Two replicas therefore never claim the same reminder.** This is the entire concurrency mechanism — no leader election, no clustered lock, no Quartz JobStore. (Satisfies the overview NFR "Stateless background work".)
2. For each claimed `id`: set `sent_at = now()` (`UPDATE reminder SET sent_at = now() WHERE id = ?` or via the managed entity) — **in the same transaction** that holds the lock, so claim + mark commit atomically. Collect the `bookingId`s.
3. Commit the transaction (releases the locks; the rows are now `sent_at != null` so no future tick reclaims them).
4. **After commit**, fire `ReminderDue(bookingId)` for each claimed reminder. Plan 4's `AFTER_SUCCESS` observer sends the email. Firing post-commit means a row is marked sent durably before we hand off to email; at-least-once email delivery (a crash between commit and fire just means a missed send for that row, acceptable for reminders) without ever sending the same reminder from two replicas.

### Task 7: Pending auto-expiry tick (feature 14) — claim expired PENDING bookings, decline, free the slot

The second piece of background work. A `@ApplicationScoped` `PendingExpiryScheduler` with its own `@Scheduled` tick that claims expired PENDING bookings with `SELECT … FOR UPDATE SKIP LOCKED`, flips them to `DECLINED`, and fires `BookingDeclined`. (Kept in a separate class from `ReminderScheduler` for single-responsibility; it could equally be a second method on the same bean — both are multi-node-safe by the same mechanism.)

**Files:**
- Create: `src/main/java/com/calit/scheduler/PendingExpiryScheduler.java`
- Modify: `src/main/resources/application.properties`
- Test: `src/test/java/com/calit/scheduler/PendingExpiryTest.java`

**Configuration:**
- `calit.approval.hold-hours` (int, default `24`) — how long a PENDING booking is held before auto-decline.

**Behavior contract for `expirePendingBookings()`:**
1. Expiry instant for a PENDING booking = `min(createdAt + holdHours, startUtc)` — a booking auto-declines after the hold window OR when its own start passes, whichever comes first (no point holding a slot past the meeting time). This is computed in SQL so the claim is a single `SELECT … FOR UPDATE SKIP LOCKED`.
2. Native claim query (parameterized hold window):
   ```sql
   SELECT id FROM booking
   WHERE status = 'PENDING'
     AND LEAST(created_at + (:holdHours * INTERVAL '1 hour'), start_utc) <= now()
   ORDER BY created_at
   FOR UPDATE SKIP LOCKED
   LIMIT 50
   ```
3. For each claimed `id`: set `status = DECLINED` **in the same transaction** (atomic claim+flip). Collect the `id`s. **Because the `booking_no_overlap_confirmed` exclusion guard is partial on `status IN ('PENDING','CONFIRMED')`, flipping to `DECLINED` drops the row from the guard — the held slot is immediately free** for a new booking, at both the app level (availability re-check ignores non-PENDING/CONFIRMED) and the DB level (the constraint no longer covers it). No explicit slot-release bookkeeping is needed.
4. After commit, fire `BookingDeclined(bookingId)` for each — Plan 4 sends the declined email, and `ReminderScheduler.onDeclined` (Task 5) deletes any (improbable) unsent reminder. A PENDING booking has no Google event yet (Plan 3 creates the event only on CONFIRMED), so there is nothing to delete in Google.
5. `@Scheduled(every = "60s")`. Same `SKIP LOCKED` guarantee: two replicas never decline the same booking.

```java
package com.calit.scheduler;

import com.calit.booking.Booking;
import com.calit.booking.BookingStatus;
import com.calit.booking.events.BookingDeclined;
import io.quarkus.narayana.jta.QuarkusTransaction;
import io.quarkus.scheduler.Scheduled;
import jakarta.enterprise.context.ApplicationScoped;
import jakarta.enterprise.event.Event;
import jakarta.inject.Inject;
import jakarta.persistence.EntityManager;
import org.eclipse.microprofile.config.inject.ConfigProperty;

import java.util.ArrayList;
import java.util.List;

/**
 * Feature 14: auto-expire approval-mode (PENDING) bookings whose hold window has
 * elapsed. Runs on EVERY replica every 60s. Multi-node-safe with NO leader: each
 * tick claims expired PENDING rows with SELECT ... FOR UPDATE SKIP LOCKED, so two
 * replicas never decline the same booking. Flipping PENDING -> DECLINED drops the
 * row from the booking_no_overlap_confirmed guard (partial on PENDING/CONFIRMED),
 * which immediately frees the held slot for a new booking.
 */
@ApplicationScoped
public class PendingExpiryScheduler {

    @ConfigProperty(name = "calit.approval.hold-hours", defaultValue = "24")
    int holdHours;

    @Inject
    EntityManager em;

    @Inject
    Event<BookingDeclined> bookingDeclinedEvent;

    @Scheduled(every = "60s")
    void expirePendingBookings() {
        List<Long> declinedIds = claimAndDeclineExpired();
        // Fire AFTER the flip transaction commits.
        for (Long id : declinedIds) {
            bookingDeclinedEvent.fire(new BookingDeclined(id));
        }
    }

    /**
     * Claims expired PENDING bookings FOR UPDATE SKIP LOCKED and flips them to DECLINED
     * in the SAME transaction. Expiry = min(createdAt + holdHours, startUtc) <= now().
     */
    List<Long> claimAndDeclineExpired() {
        return QuarkusTransaction.requiringNew().call(() -> {
            @SuppressWarnings("unchecked")
            List<Number> ids = em.createNativeQuery(
                    "SELECT id FROM booking "
                            + "WHERE status = 'PENDING' "
                            + "AND LEAST(created_at + (:holdHours * INTERVAL '1 hour'), start_utc) <= now() "
                            + "ORDER BY created_at "
                            + "FOR UPDATE SKIP LOCKED "
                            + "LIMIT 50")
                    .setParameter("holdHours", holdHours)
                    .getResultList();

            List<Long> declinedIds = new ArrayList<>();
            for (Number n : ids) {
                Long id = n.longValue();
                Booking b = Booking.findById(id);
                b.status = BookingStatus.DECLINED;   // flipped within the lock-holding transaction
                declinedIds.add(id);
            }
            return declinedIds;
        });
    }
}
```
